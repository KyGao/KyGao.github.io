## C++基础

### 虚函数的实现机制
虚函数的作用主要是实现了多态的机制即动态联编
- **动态联编**
  首先函数名联编是指将源代码中的函数调用为特定代码块，在C++中，由于函数重载，需要根据**函数名**和**函数参数**一起来确定使用哪个函数块，在编译过程中实现这个过程的联编叫做**静态联编**，但是C++还提供了一个情况，就是函数名和函数参数都一样的情况也可以有不同功能，这就必须要在程序运行时才能实现，就是**动态联编**，在这时，编译器必须生成能够在程序运行时选择正确的虚方法的代码。注意，动态绑定要求采用 `指针->函数()` 或 `引用变量.函数()` 的方式调用C++类中的虚函数才能实现。

**答**：主要是依靠**虚函数表**和**虚表指针**实现的，每个类使用一个虚函数表，每个类对象用一个虚表指针。比如基类对象包含一个虚表指针，指向基类所有虚函数的地址表，派生类对象也有一个虚表指针，指向派生类所有虚函数的地址表。如果派生类重写了基类的虚方法，派生类的虚函数表将保存重写的虚函数的地址；如果派生类没有重写基类的虚方法，即将继承基类的虚方法，那么派生类的虚函数表中将保存基类未被重写的虚函数的地址。

### 构造函数/析构函数可不可以virtual
**答**：**构造函数**：C++之父对这个问题做过回答，大概含义是说：虚函数调用是在部分信息下完成工作的机制，允许我们只知道接口而不知道对象的类型。然而，要创建一个对象，就一定要知道对象的完整信息，特别是想要创建对象的确切类型，所以构造函数不应被定义为虚函数。所以结论是这样的：构造函数任何时候都不能声明为虚函数，并且不能在构造函数内调用虚函数。具体来看，虚函数的调用需要找到虚函数表，虚函数表是存储在内存空间的，但是在调用构造函数时，对象还未被实例化，也就没有该对象的内存空间，也就无法找到虚函数表，所以说虚函数表是在构造对象之后才建立的，也就不可以 virtual。
**析构函数**却有点不一样，基类的析构函数通常需要 virtual 虚析构函数，目的是防止内存泄漏。假如基类采用的是非虚析构函数，那么在派生类的析构函数释放派生类对象内存的时候，删除基类指针指向派生类对象时不会触发动态绑定，即只会调用基类的析构函数而不会调用派生类析构函数，所以派生类申请的空间就得不到释放而产生内存泄漏。但是依然不能在析构函数中调用虚函数，因为析构一旦开始执行，先是派生类成员就会呈现未定义值，进入基类析构后，对象就成为了一个基类对象，动态调用也没有意义了。

### 继承体系下函数的调用
![C++继承体系](https://img-blog.csdn.net/20150706123206122?watermark/2/text/aHR0cDovL2Jsb2cuY3Nkbi5uZXQv/font/5a6L5L2T/fontsize/400/fill/I0JBQkFCMA==/dissolve/70/gravity/Center)

**答**：实函数和虚函数调用机制不相同，静态调用时用的是类的编译信息，实函数与虚函数的静态调用时采用，动态调用只能是虚函数，用的是虚函数表的函数信息。在单继承体制中，子类的虚函数会把基类的虚函数表拷贝过来，同时子类的虚函数会覆盖同名的虚函数，对于这种虚函数，就可以静态调用实现基类的功能或者动态调用实现派生类重载的功能了。
从另一个角度来看，有三种继承方式：公有，受保护和私有。public继承时，派生类继承基类的公有和保护成员，且成员属性不变，基类中类外能访问的在派生类中也能类外访问；protected继承时，也是继承基类的公有和保护成员，但是成员属性都变为保护，即类外不能访问成员，只能通过成员函数访问；private继承时，基类的公有和保护成员均变为派生类的私有成员，无法通过子类访问。同时要注意的一点是，友元关系不能继承。

### 内存分配、内存对齐、序列化等
- **堆栈对比**
  栈是系统提供的功能，由编译器自动分配释放，优点是快速高效，缺点是有限制，数据不灵活；堆是函数库提供的功能，一般由程序员分配释放优点是灵活方便，数据适应面广泛，但是效率有一定降低
- **内存分配**
  程序占用的内存一般分为以下几个部分：栈区，堆区，全局区（也成为静态区），文字常量区和程序代码区
- **内存对齐**
  现代计算机中内存空间都是按byte划分的，通常会要求数据的首地址的值是某个数的 k 倍，这样可以匹配计算机的字长，加快数据存储的速度，而不用花时间计算某个数据的哪几位是int哪几位是char
- **序列化**
  序列化就是将对象变成字节流的形式传出去，反序列化就是从字节流恢复成原来的对象。实现存储和传输数据时的必需操作，比如可以序列化一个对象，然后使用 HTTP 通过 Internet 在客户端和服务器之间传输该对象。常用方法如Google Protocol Buffers 的 protobuf等。

### Static关键字的作用
有五处使用场景：
- **全局静态变量**
  全局静态变量加上 static 就变成了全局静态变量，作用域：定义之处开始，到文件结尾，存在于静态存储区（文件外不可见）
- **局部静态变量**
  局部静态变量加上 static 就变成了局部静态变量，作用域：定义之处开始，到定义他的函数或语句块结束，只初始化一次（放在 while 里不会重复赋值）
- **静态函数**
  函数返回类型前加上 static 就变成了静态函数，将函数限制在该文件内才有用，从而不会和其他文件的同名函数发生冲突
- **类的静态数据成员**
  静态数据成员是类的所有对象中共享的成员，而不是某个特定对象的成员。优点是不会破坏类的隐藏原则，保证了安全性
- **类的静态成员函数**
  类的静态成员函数和静态数据成员一样，他们都属于类的静态成员，都不是对象成员。特点：静态成员函数实现时不能引用类的非静态成员，只能引用静态成员（毕竟是基于静态成员实现的方法），如果静态成员函数中要引用非静态成员时，可通过对象来引用

### 虚函数和纯虚函数的区别
- **抽象类**：包含纯虚函数的类
  这种类不能声明对象，只是作为基类为派生类服务。除非在派生类中实现了基类中所有纯虚函数，否则·派生类也变成了抽象类，不能实例化对象。
**答**：虚函数可以直接使用，也可以在子类中重载以多态的形式调用，而纯虚函数在基类中只有声明没有定义，所以只能在子类中实现了该函数才可以以多态的形式调用。这么来看的话，虚函数在子类中可以不被重载，但是纯虚函数必须在子类中实现。
### 虚函数以及构造函数执行顺序的一些特性
**答**：构造函数时先构造基类，再构造派生类；析构函数时先析构派生类，再析构基类。

### struct/class 以及 union 内存对齐的四个原则
**答**：该四原则适用于 struct / class / union ：
  1. 数据成员对齐规则：结构或联合的数据成员，第一个数据成员放在 offset 为 0 的地方，以后每个数据成员存储的起始位置要从该成员大小或者成员的子成员大小（子成员可能是数组、结构体等）的整数倍开始（比如 int 在32位机为4字节，则要从4的整数倍地址开始存储）
  2. 结构体作为成员：如果一个结构里有某些结构体的成员，则结构体成员要从其内部“最宽基本类型成员”的整数倍地址开始存储（struct a里有struct b，b里有char, int, double等元素，那么b应该从8的整数倍开始存储）
  3. 收尾工作：结构体的总大小，也就是sizeof的结果，必须是其内部最大成员的“最宽基本类型成员”的整数倍，不足的要补齐
  4. sizeof(union)，以结构里面size最大元素为union的size。因为在某一时刻，union只有一个成员真正存储于该地址

### 如何让.h文件不被重复引用
**答**：可以使用条件编译，就是
  ```c
  #ifndef _HEADERNAME_H
  #DEFINE _HEADERNAME_H
  ...
  #endif
  ```
  但是无法保证变量不被重复定义（可以用定义成静态变量的方法）
  另一种方法是 `#pragmaonce` 但是这个方法可能会有跨平台的问题，因为并不是所有编译器都支持这个操作

## 操作系统
### 什么是操作系统
**答**: 四句话
1. 操作系统是管理计算机硬件与软件资源的程序，是计算机系统的内核与基石。
2. 操作系统本质上是运行在计算机上的软件程序。
3. 操作系统为用户提供一个与系统交互的操作界面。
4. 操作系统分内核和外壳，内核负责管理系统进程、内存、设备驱动程序等操作硬件的程序，外壳指的是围绕内核的应用程序

### 什么是系统调用
- **用户态和系统态**: 根据进程访问资源的特点，我们可以把进程在系统上的运行分为两个级别。**用户态**运行的进程或可以直接读取用户程序的数据，**系统态**运行的进程或程序几乎可以访问计算机的任何资源，不受限制。

**答**: 我们运行的程序基本都是运行在用户态，如果我们需要调用系统态级别的资源时，比如文件管理、进程控制、内存管理等，都必须通过系统调用方式向操作系统提出服务请求，并由操作系统代为完成。

### 大致可以分为哪几类系统调用
**答**: 
1. 设备管理。设备的请求、释放、启动等
2. 文件管理。文件的读、写、创建删除等
3. 进程控制。进程的创建、撤销、阻塞、唤醒等
4. 进程通信。进程间消息传递
5. 内存管理。内存的分配回收，获取作业占用内存等

### 进程和线程的区别
**答**: 线程是进程划分成的更小的运行单位，一个进程在其执行的过程中可以产生多个线程。线程和进程最大不同在于基本上各进程是独立的，而各线程则不一定，因为同一进程中的线程极有可能相互影响。线程执行开销小，但不利于资源的管理和保护，而进程正好相反。
